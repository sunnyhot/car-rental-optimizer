# 神州跨城查询限流恢复设计

## 背景与根因

神州指定车型搜索会扫描搜索半径内的候选城市。近期实现把城市上限从 20 提升到 60，并使用最多 6 个并发任务执行每个城市的 `deptList -> chooseCar` 请求链，以避免 500 km 搜索因串行执行而超时。

官方网关会在短时间请求过密时返回“访问频繁，请稍后再试”。当前客户端把该业务响应当作普通不可重试错误，同时立即补入下一座城市，导致请求压力不下降；当目标车型没有从其余城市返回时，界面最终显示“神州 API 部分查询失败”。

## 目标

- 识别神州网关的限流响应，并在有限时间内自动恢复。
- 保留最多 60 个候选城市的搜索覆盖，避免重新漏掉较远城市的低价车源。
- 避免无限重试，并继续服从平台级搜索超时和任务取消。
- 不改变一嗨查询、候选排序、价格计算、登录 Cookie 或确认页费用行为。

## 方案

为神州城市查询增加共享的请求节流器，并将城市查询并发从 6 降低到更保守的 3：

1. 每次 `deptList` 或 `chooseCar` 请求前，先通过节流器取得请求时机，使同一轮神州城市扫描的请求不会瞬间成批发出。
2. 将包含“访问频繁”或等价限流含义的网关业务响应标记为可重试限流错误。
3. 命中限流后更新共享冷却窗口，使其他城市任务也暂停发送新请求，而不是各自继续压测网关。
4. 当前请求进行有限次数重试，使用递增退避；超过重试次数后保留原始平台错误，由现有错误聚合逻辑展示。
5. 普通业务错误、解码错误和非重试网络错误不重试；现有移动域名到桌面域名的传输故障兜底保持不变。

节流器只作用于高扇出的跨城市 `deptList` 和 `chooseCar` 调用。城市列表请求与登录后的确认页费用补全维持现状，避免扩大改动面。

## 数据流与错误处理

```text
城市任务
  -> 等待共享发送间隔/冷却窗口
  -> 调用 deptList 或 chooseCar
  -> 成功：继续该城市请求链
  -> 限流：登记全局冷却，退避后有限重试
  -> 其他失败：立即返回城市错误
```

若部分城市最终失败但其他城市返回匹配车型，继续采用成功结果。只有没有任何匹配车型且存在查询错误时，才沿用当前“部分查询失败”状态。

## 测试与验证

- 单元测试：限流文案可被准确识别，普通业务错误不会误判。
- 异步单元测试：首次限流后会等待并重试，后续成功结果可返回；超过上限后返回原始错误。
- 源码/配置回归：城市扫描保持有界并发，并使用共享节流器。
- 运行相关测试目标，再运行完整 `swift test` 与 `swift build`。
- 若本机网络条件允许，使用截图中的搜索条件做手工冒烟验证，确认不再立即出现“访问频繁”。

## 边界

本修复不会尝试绕过平台验证或无限提高请求吞吐；若平台持续限流超过有限重试窗口，仍会向用户显示真实失败原因。退避等待必须响应 Swift 任务取消，确保平台级超时能够终止搜索。
