# Decision Comparison and Full UI Refactor Design

## Goal

将租车比价助手重构为统一的 macOS 决策工作台，并新增 2–4 个候选方案的原位横向比较能力。用户应能在不离开当前搜索上下文的情况下比较真实报价、到店路线、车型信息和数据可信度，同时在比价、价格监控和登录弹窗之间获得一致、清晰且可访问的视觉体验。

本次采用全量 UI 重构路线。重构覆盖应用壳层、比价工作区、原位比较矩阵、价格监控工作区和主要弹窗，但不改变官方平台 API、候选排序、价格计算、登录 Cookie、价格监控存储格式或发布机制。

## Product Principles

- 真实数据优先：比较内容只来自官方平台报价、现有路线估算和明确标注来源的车型资料。
- 决策而非展示：界面帮助用户理解成本、距离、车型和可信度的取舍，不制造单一且误导性的“综合赢家”。
- 上下文连续：进入比较、切换导航或查看监控时尽量保留当前搜索条件、筛选和选择。
- 状态可解释：加载、缺失、失败、登录、验证码和历史结果都使用明确文字与恢复操作。
- 原生且克制：使用 SwiftUI、SF Symbols 和系统字体，不引入第三方 UI 依赖或持续抢眼的装饰动效。

## Scope

### Included

- 统一应用壳层与主导航。
- 比价工作台三栏布局的完整视觉与交互重构。
- 候选 2–4 项选择、底部选择状态和原位比较矩阵。
- 完整比较维度与“只看差异”过滤。
- 已选车型资料的按需异步加载与独立失败处理。
- 价格监控从大型弹窗整合为主导航工作区。
- 创建监控、一嗨登录、神州登录和通用状态弹窗的视觉统一。
- 浅色/深色模式、键盘焦点、辅助功能标签和减少动态效果支持。
- 比较状态、展示模型和关键 UI 状态的自动化测试。

### Excluded

- 新增租车平台或更换现有平台接口。
- 修改候选排序、价格计算、车型匹配或路线估算口径。
- 修改登录 Cookie 或监控数据的持久化格式。
- 保存比较历史、收藏比较组、导出或分享比较结果。
- 引入移动端布局、第三方 UI 框架或外部字体。
- Developer ID 签名、公证、发布流程或版本发布。

## Information Architecture

### Application Shell

`MainView` 从单一三栏容器升级为统一应用壳层：

- 左侧紧凑导航栏提供“比价工作台”和“价格监控”两个主目的地。
- 顶部状态带集中展示官方 API、搜索进度、上次成功时间和监控健康状态。
- 主内容区根据导航选择展示比价工作区或监控工作区。
- 一嗨/神州登录、创建监控和短时确认流程继续使用 sheet，避免把短流程提升为主导航页面。

菜单命令与通知继续工作：重新比较切回比价工作台，打开监控中心切换至监控工作区，到期巡查继续触发现有监控命令。

### Search Workspace

常规状态保留三栏心智模型：

- 左栏为行程配置器，包含位置、日期、车型、半径、还车方式、平台选择和平台状态。
- 中栏为候选浏览器，包含搜索摘要、排序、筛选、候选卡片和比较选择状态。
- 右栏为单个候选的决策路径，包含总价、费用拆分、到店路线、车型资料、可信度、风险和操作。

进入比较模式后，左栏保持不变，中栏与右栏合并为横向比较矩阵。退出比较后恢复原筛选、排序、滚动上下文和当前候选。

### Monitor Workspace

监控中心不再作为大型 sheet 展示，而是成为主导航工作区：

- 左侧展示可筛选、按注意力优先排序的监控列表。
- 右侧展示当前监控的摘要、趋势、事件、匹配条件和历史快照。
- 顶部健康状态带展示需处理、近期降价、到期巡查和后台巡查状态。
- 创建监控继续使用稳定尺寸的 sheet。

## Component Boundaries

### Existing Ownership

- `SearchViewModel` 继续负责搜索请求、平台状态、真实候选、筛选、排序和当前单项选择。
- `MonitorCenterViewModel` 继续负责监控列表、健康摘要、巡查和批量操作。
- `LiveRentalSearchService`、领域模型、地图服务、登录会话与监控存储保持现有职责。

### New Comparison Ownership

新增 `ComparisonWorkspaceViewModel`，专门负责：

- 保存当前搜索结果中的 0–4 个候选 ID。
- 判断是否满足至少 2 项的比较门槛。
- 进入和退出比较模式。
- 控制“只看差异”。
- 为已选候选加载、缓存和取消车型资料请求。
- 将单个候选资料加载失败隔离在对应列。
- 在新搜索开始时清空旧比较状态。

比较展示层使用独立的 presentation model，将现有 `Recommendation`、报价可信度和 `VehicleInsight` 转换为分组矩阵行。差异判断和最佳值标记应为可单元测试的纯逻辑，不写在 SwiftUI `body` 中。

`WorkbenchStyle` 继续作为共享设计系统入口。若全量重构使文件过大，应按职责拆分为 token、surface、status、navigation 和 comparison 组件文件，同时维持单一语义 token 来源。

## Comparison Interaction

### Selection

- 候选卡片始终显示轻量复选框，避免把比较入口隐藏在悬停或菜单中。
- 第 2 项被选中后，中栏底部出现比较状态栏和“开始对比”操作。
- 最少 2 项、最多 4 项；达到上限后未选复选框进入禁用状态并说明原因。
- 筛选和排序变化不删除已选候选，即使某项暂时被筛选隐藏。
- 开始新的搜索时清空比较选择，避免跨搜索混合不同时间或条件下的报价。

### Matrix

矩阵使用候选作为列、比较项作为行，并按以下顺序分组：

1. 决策摘要：总成本、平台、车型、门店和当前推荐状态。
2. 费用：租车费、平台费、保险费、异店费、到店成本。
3. 门店与路线：距离、地址、营业时间、打车方案和公共交通方案。
4. 车型：类别、匹配度、尺寸、座位、能源信息和常用配置。
5. 可信度与风险：费用完整度、资料来源、跨城、登录复核和其他警告。

默认展示所有适用维度。“只看差异”隐藏所有已选候选值均相同的行，但始终保留分组标题、候选标题和核心总成本行。

最佳值按维度分别标记：最低总成本、最低租车费、较近门店、较短路线和更完整报价各自独立，不计算新的综合分数。缺失车型资料显示“未确认”，不能解释为不具备该配置。

每个候选列提供移除、设为当前方案、创建价格监控和打开官方页面操作。移除后只剩 1 项时退出矩阵、保留剩余选择，并提示用户再选择一个候选。

### Asynchronous Vehicle Insight

- 价格、路线和可信度使用当前候选数据立即渲染。
- 车型资料只对已选候选按需加载，不为所有搜索结果预取网络资料。
- 每列独立显示加载、已缓存、失败或未确认状态。
- 单列失败不阻塞矩阵其余内容，也不使比较整体失败。
- 候选被移除或新搜索开始时取消不再需要的请求。

## Visual System: Route Blueprint

### Product-Specific Direction

视觉方向命名为“路线蓝图”。其唯一签名元素是一条表示“位置 → 门店 → 报价 → 决策”的路线轨迹。轨迹只用于应用状态带、路线卡和比较流程，不作为无意义装饰散布到所有卡片。

### Palette

- `BlueprintMist`：浅色模式主背景，参考色 `#EAF1F5`。
- `RouteInk`：深层标题和深色表面，参考色 `#173953`。
- `DecisionBlue`：主操作、选中态和比较焦点，参考色 `#2469D3`。
- `SignalTeal`：路线、实时成功和完整报价，参考色 `#0D9B96`。
- `RiskAmber`：登录、待复核和可恢复警告，参考色 `#D98A22`。
- 失败继续使用系统语义红色；所有颜色通过 AppKit adaptive color 适配浅色和深色模式。

### Typography and Surfaces

- 界面文字使用 SF Pro 系统字体，保持原生渲染和动态可读性。
- 价格、距离、时间和百分比使用 monospaced digit，不把全界面改为等宽字体。
- 卡片圆角保持 6–10px，选中态使用结构边线和列背景，不使用大面积高饱和填充。
- 阴影仅区分层级；深色模式主要依赖表面明度与边线，不使用持续霓虹发光。

### Motion

- 查询、巡查和车型资料加载使用状态轨迹的短周期运动。
- 工作区切换、结果出现和比较模式切换使用 150–300ms 的 opacity/offset 过渡。
- 不让动画阻塞点击、输入或滚动。
- `accessibilityReduceMotion` 开启时移除循环、位移和错峰，只保留颜色与内容状态变化。

## Page-Level Design

### Trip Configuration

左栏按“行程坐标 → 租期 → 车辆与范围 → 平台”组织。地址候选、日期范围、车型建议和平台恢复操作保持现有业务规则，但共享一致的字段容器、焦点边线和状态行。主操作文案保持“开始比较”，加载时展示当前搜索阶段。

### Candidate Browser

候选卡以总成本、车型、平台、门店距离和可信度为首屏扫描信息。筛选和排序整合在紧凑工具条中，比较选择状态与普通单项选择在结构上可区分。历史结果必须继续显示明确的时间和过期提示。

### Decision Path

右栏按“推荐总成本 → 费用 → 到店路线 → 车型资料 → 风险 → 操作”组织。路线轨迹连接起点、出行方式和门店，帮助用户理解到店成本的来源，而不是装饰性流程图。

### Monitor Workspace

监控列表强化状态、下次巡查、最近价格变化和需处理原因。详情区域保持现有趋势和事件数据口径，只重新组织层级、操作位置和空状态。批量暂停、恢复与巡查仍作用于当前筛选集合。

### Sheets and Login Views

创建监控、一嗨登录和神州登录共用统一 sheet 壳层、标题区、状态行和操作层级。WebView、Cookie 保存、验证码处理和刷新/重置行为不变。状态文案变长时 sheet 尺寸保持稳定，不产生明显布局跳动。

## Error, Empty, and Historical States

- 查询加载态按位置解析、平台读取、路线计算和完成阶段展示。
- 无结果时展示各平台的真实返回状态与可执行恢复动作，不生成推测价格。
- 登录、验证码、平台不可用、解析失败和网络失败使用现有 typed status 映射。
- 新搜索失败时保留上次成功结果，并在候选区和详情区持续标记为历史结果。
- 车型资料失败只影响对应比较列，提供重试或“未确认”降级。
- 监控无选择、无快照、连续失败和失败后恢复均使用专用空状态或事件状态。

## Accessibility and Window Behavior

- 图标按钮必须提供可见文本、help 或 accessibility label。
- 选择、最佳值和风险不能只靠颜色表达，必须同时使用文字、图标或边线结构。
- 保留系统键盘焦点行为，并为主导航、候选选择、比较进入/退出和矩阵操作提供合理 tab 顺序。
- 主窗口最低宽度维持约 1280px；新增导航使用紧凑宽度，不以压缩文本换取空间。
- 长门店名、车型名和风险文案允许换行或截断，但价格、可信度和关键操作不能被遮挡。
- 浅色、深色、高对比和减少动态模式均应可用。

## Testing and Verification

### Automated Tests

- `ComparisonWorkspaceViewModel`：选择下限、上限、移除、重复选择、新搜索重置和筛选隐藏保留。
- 比较 presentation：分组顺序、金额格式、最佳值标记、缺失值和“只看差异”。
- 异步资料：独立成功/失败、缓存复用、取消和过期响应不覆盖新状态。
- 导航协调：菜单通知切换至正确工作区且现有搜索、监控命令仍执行。
- 现有 `SearchViewModel`、监控、登录、领域排序和价格计算测试保持通过。
- 运行 `swift build` 与 `swift test`。

### App Verification

- 使用 `scripts/build-app.sh` 构建 App Bundle。
- 使用 `scripts/verify-launch.sh build/租车比价助手.app` 做启动验证。
- 检查比价常规态、2/4 项选择、比较矩阵、只看差异、车型资料加载/失败和退出恢复。
- 检查监控工作区、创建监控、一嗨登录和神州登录。
- 检查浅色/深色、最小窗口、键盘焦点和减少动态效果。

## Delivery Sequence

实现计划应按以下可验证阶段拆分，每阶段保持可构建：

1. 路线蓝图 token、共享组件和应用壳层。
2. 比较状态模型、纯展示逻辑和单元测试。
3. 候选多选、比较状态栏和原位矩阵。
4. 已选车型资料的异步加载与降级状态。
5. 比价工作台三栏的完整页面重构。
6. 监控中心整合为主工作区。
7. 登录、创建监控和通用弹窗统一。
8. 自动化回归、App Bundle 启动和完整视觉验收。

## Success Criteria

- 用户能从一次搜索中选择 2–4 个真实候选，并在当前工作台原位比较。
- 用户能查看完整维度或只看差异，并明确区分成本、距离、报价完整度和车型信息的不同优势。
- 车型资料缺失或单列加载失败不会阻止价格和路线比较。
- 比价、监控和登录流程共享一致的路线蓝图视觉语言。
- 所有现有真实 API、排序、监控和登录行为保持兼容。
- 自动化测试、构建、App Bundle 启动与关键视觉状态验证全部通过。
